为什么 MIN(storage_hit) 一致？

各 rank 命中可能不同（host 内存截断、L3 视图差异）。MIN 取最长公共可命中前缀 → 所有 rank 抓取范围一致，不会各抓不同长度。

为什么 MIN(completed_tokens) 一致？

即便抓取范围一致，实际逐页加载仍可能部分失败（page_get 返回 n≠batch）。MIN 只提交所有 rank 都成功落盘的最长公共前缀 → 写入 host tree 的长度逐 rank 相同。

为什么不会 hang？

每个 storage batch 都产生且仅产生一个 PrefetchAck（即使出错也照常产生）→ 每个 rank 参与的 reduce 次数严格相等，collective 一一对齐。两个 MIN 一起保证：插入 host tree 的前缀逐 rank 完全相同 → 树一致。

两个 MIN 同步点（组1 命中数、组2 完成数）+ 每 batch 恒定 1 个 ack，共同保证 PP 各 rank 的 host radix tree 严格一致。
每个 storage batch 在 _page_transfer 里恒产 1 个 PrefetchAck；prefetch_sync_thread 对每个 ack 在组2 做一次 all_reduce(MIN)。所以 ack 数 = batch 数 = 组2 collective 次数，必须逐 rank 相等。
ack 已产出（参与本轮 reduce） barrier 凑齐 3/3 → 通过 已到达，等待缺席方 缺失 ack → 永远等不到
◆ all_reduce(MIN) @ 组2（prefetch_sync_groups2）· 每个 ack 一次 barrier
选择场景：每 batch 恒 1 ack → 次数对齐、安全；出错就 break → ack 缺一个 → 组2 reduce 错位 → hang。
